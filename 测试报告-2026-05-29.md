## 发现的 Bug

### BUG-1 🔴 云中转开关形同虚设

**文件**：`ConnectionManager.kt` 第 287-313 行  
**严重度**：高 — 解释场景 2 的所有实验现象  

```kotlin
fun loadSavedConfig() {
    val cloudEnabled = prefs.getBoolean("cloud_enabled", false)  // ← 读取了但从未使用
    ...
    if (cloudServerList.isNotEmpty()) connectCloud(cloudServerList, lastPin)  // ← 只看serverList，不检查cloudEnabled
}
```

**`scheduleCloudReconnect` 也不检查 `cloudEnabled`**（第 868 行）：
```kotlin
private fun scheduleCloudReconnect() {
    if (!autoReconnect || lastPin.isEmpty()) return  // 只检查autoReconnect
    ...
}
```

**后果**：只要手机的 `cloud_servers` 持久化里还有服务器地址，无论云中转开关是开是关，`loadSavedConfig`、`scheduleReconnect`、`scheduleCloudReconnect` 都会尝试连接云端。"云开关"只是一个 UI 状态，完全没有控制连接行为。

**影响场景**：场景 2 实验 1-5（所有实验手机都自动连云端，包括开关关的时候）。

### BUG-2 🟡 云端 phone_hello 注册的设备可能用旧 cloudWs

**文件**：`main.js` `connectCloudServer` 函数  
**严重度**：中 — 时机依赖，解释了场景 1  

```javascript
function connectCloudServer(targetServerUrl, onResult) {
  if (cloudWs) {
    try { cloudWs.close(); } catch (e) {}
    cloudWs = null;   // ← 先把 cloudWs 置 null
  }
  // ... 然后才创建新连接
  cloudWs = new WebSocket(serverUrl);
```

当云端发生重连时，`cloudWs` 变量被短暂清空为 `null`。如果此时恰好有 cloudWs 的旧 message handler 还有残留消息，调用 `registerDevice(pin, { cloudWs: cloudWs, ... })` 时 `cloudWs` 是 `null`，设备被注册为 `device.cloudWs = null` → `cloudOk = false` → `getActiveDevice()` 返回 null。

**影响场景**：场景 1（手机退出再重开，PC 卡在 LAN）。

### BUG-4 🟢 scheduleCloudReconnect 无最大尝试次数上限

**文件**：`ConnectionManager.kt` 第 868-883 行  
**严重度**：低 — 手机永久重试云端  

```kotlin
private fun scheduleCloudReconnect() {
    cloudReconnectAttempts++   // 无限递增
    val delayMs = getReconnectDelay(cloudReconnectAttempts)
    // ← 没有 if (cloudReconnectAttempts >= MAX) return
```

`scheduleReconnect`（LAN 重连）有 30 次上限检查，但 `scheduleCloudReconnect` 没有。云端 relay 永久挂掉时手机每 5 分钟重试一次、永不停。

### BUG-5 🟢 插件拨号走 WebSocket 不经过自动唤醒逻辑

**文件**：`main.js` 第 1448-1460 行  
**严重度**：低 — 插件拨号失败时直接返回错误  

```javascript
// 插件发送拨号命令
if (msg.type === 'dial' && ws.isPlugin) {
    const active = PhoneConnectionManager.getActiveDevice();
    if (!active) {
        ws.send(JSON.stringify({ type: 'dial_fail', reason: '手机未连接' }));
        return;  // ← 直接返回错误，不触发自动唤醒
    }
```

浏览器的插件拨号走了 WebSocket 直连路径，绕过了 IPC `dial` 处理器的自动唤醒和排队逻辑。手机不在线时插件直接报"手机未连接"，不会触发 `_tryWakePhone` 或 `queueDial`。

## 逐场景测试结果

### 场景 1：手机退出重开，PC 卡 LAN

| 步骤 | 代码路径 | 结果 |
|------|---------|------|
| 手机退出 → LAN WS close | `ws.on('close')` → `removeDevice('lan')` | ✅ device.stale=true, device.cloudWs 仍存在 |
| 3s 后手机重开 → connectCloud | `loadSavedConfig` → `connectCloud` | ✅ 云端 phone_hello 到达 PC |
| PC 接收 phone_hello | `registerDevice(pin, {cloudWs, isCloud:true})` | ⚠️ 若此时 cloudWs 被置 null（BUG-2），设备注册后 cloudWs=null |
| 用户点拨号 | `getActiveDevice()` → `cloudOk = cloudWs.readyState === 1` | ❌ cloudWs 为 null → cloudOk=false → 返回 null |
| 点 ☁ 重连按钮 | `connectCloudServersFromList` → 新 cloudWs | ✅ 重建后设备恢复正常 |

**结论**：⚠️ 条件 Bug，取决于云端是否在 3 秒窗口内触发重连。

### 场景 2：断开后秒速重连

| 实验 | 预置条件 | 根因 | 结果 |
|------|---------|------|------|

## Bug 汇总

| # | 严重度 | 文件 | 问题 |
|---|--------|------|------|
| 1 | 🔴 高 | ConnectionManager.kt | 云中转开关不控制重连行为 |
| 2 | 🟡 中 | main.js | connectCloudServer 短暂清空 cloudWs 导致注册失败 |

## 非 Bug 观察项

| 项目 | 描述 |
|------|------|
| 网络抖动计数器重置 | NetworkMonitor 每次都 reset reconnectAttempts=0，可能永远达不到 30 次上限 |
